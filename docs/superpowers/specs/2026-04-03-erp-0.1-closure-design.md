# ERP 0.1 Closure Design

Date: 2026-04-03
Status: approved-for-planning
Scope: close Layer 0 item 0.1 by proving and implementing a real Sankhya-backed vertical slice for `products`, `prices`, and `inventory`

## Purpose

ERP 0.1 is not complete until MetalShopping can:

- connect to the real Sankhya/Oracle source
- prove the source truth for `products`, `prices`, and `inventory`
- ingest those feeds into raw ERP state
- normalize them into MetalShopping language
- promote them into canonical modules with deterministic review gates
- replay the same behavior in CI through fixtures

This design closes the gap left by the current partial connector/runtime implementation.

## Non-goals

This tranche does not:

- close all remaining ERP entity types beyond `products`, `prices`, and `inventory`
- assume incremental sync before source change markers are proven
- hardcode ERP-specific naming into canonical pricing or inventory semantics
- force auxiliary product flags into taxonomy or canonical status prematurely

## Approach

Recommended approach: raw-first discovery, then one canonical vertical slice.

Why:

- the main risk is modeling Sankhya incorrectly, not implementation speed
- `TGFTAB` and `TGFNTA` may materially change the right price-history design
- pricing and inventory already require governed normalization rather than direct ERP semantics

Execution order:

1. local-only read-only Oracle discovery workbook
2. source-truth verification for `products`, `prices`, and `inventory`
3. `TGFTAB` and `TGFNTA` discovery for effective-date and price-history semantics
4. canonical schema validation, with user approval required before any schema change
5. vertical-slice implementation for raw ingestion, governed mapping, review gating, and canonical promotion
6. bounded live smoke, then full snapshot validation

## Architecture

ERP 0.1 closes through two phases.

### Phase 1: source discovery and truth capture

This phase is local-only and read-only.

Outputs:

- Oracle connection form resolved, including service/SID discovery if required
- read-only discovery workbook for:
  - `TGFPRO`
  - `TGFEST`
  - `TGFEXC`
  - `TGFTAB`
  - `TGFNTA`
- bounded sample queries that prove actual source meaning
- captured fixtures from proven queries for reproducible tests

The workbook is not committed until mappings are proven correct.

### Phase 2: canonical ingestion and promotion

This phase implements the vertical slice.

Outputs:

- raw ERP snapshot ingestion for `products`, `prices`, and `inventory`
- governed normalization into MetalShopping canonical language
- review-gated promotion into canonical modules
- bounded live smoke followed by full snapshot validation

## Extraction model

This tranche is snapshot-first.

Rules:

- use full snapshots for `products`, `prices`, and `inventory`
- do not adopt incremental watermarks until the source fields are proven trustworthy
- start live validation with a bounded sample of SKUs
- after mappings are proven, run a full snapshot

Reasoning:

- a bad incremental key silently loses updates
- snapshot-first is slower but safer while source semantics are still being verified

## Canonical product model

### Identity

MetalShopping canonical product identity for this tranche includes:

- ERP product ID as the primary source anchor
- `EAN` as an optional canonical secondary identifier
- manufacturer `REFERENCE` as an optional canonical secondary identifier

Rules:

- duplicate `EAN` triggers review-required blocking
- duplicate manufacturer `REFERENCE` triggers review-required blocking
- the ERP product ID remains the authoritative sync anchor for the connector

### Core fields

Normalized now:

- product name
- product description
- brand
- canonical product status
- taxonomy assignments for the main commercial hierarchy

Not used for now:

- manufacturer stays empty unless a trustworthy source is later discovered

### Taxonomy

Taxonomy is the canonical model for merchant-specific commercial hierarchy.

Rules:

- map only the main commercial hierarchy into taxonomy
- do not force auxiliary ERP classifications into the taxonomy tree
- preserve source lineage for every taxonomy assignment
- keep taxonomy semantically meaningful through node/assignment role, not only generic tree placement

This design assumes the existing MetalShopping taxonomy system remains the owner of category/group/subgroup-like semantics.

### Product status

Canonical status vocabulary for this tranche:

- `active`
- `inactive`
- `discontinued`
- `out_of_line`
- `unknown`

Rules:

- ERP-native statuses are translated into this canonical set
- raw source values are preserved for lineage
- auxiliary flags such as competitiveness are not part of canonical status for now

### Auxiliary product flags

Fields such as:

- `AD_COMPETITIVO`
- `AD_CLASSIFIC`
- substitution-related markers

remain raw-only until their cross-ERP business meaning is understood well enough to justify canonical treatment.

## Canonical pricing model

Pricing is multi-context by design.

Rules:

- do not collapse all prices into a single current price
- import all discovered commercial price tables/contexts
- preserve source Sankhya table identity and names for lineage
- normalize source pricing into MetalShopping-owned canonical price contexts
- use governed mapping from source table/context to canonical context
- allow seeded defaults, but not permanent hardcoded mappings

### Price history

`TGFTAB` and `TGFNTA` must be investigated as part of this tranche to determine:

- table identity semantics
- effective-date semantics
- version/history behavior
- whether defensible historical price import/backfill exists

Decision rule:

- if Sankhya exposes a defensible historical source, use it
- if not, MetalShopping starts authoritative price history from first sync onward

Defensible means the history can be traced to raw authoritative source tables or logs, not inferred from aggregates or guessed retroactively.

## Canonical inventory model

Inventory is raw-first and exact at ingestion time.

Rules:

- store the raw Sankhya inventory position exactly as reported
- preserve negative inventory values
- do not clamp or reinterpret quantities during raw sync
- keep source Sankhya company/local identities for lineage
- normalize into MetalShopping-owned canonical stock-location contexts
- use governed mapping from source company/local to canonical stock location

Derived concepts such as `available`, `sellable`, or other policy-dependent interpretations remain separate from the raw sync contract unless explicitly backed by authoritative source fields.

## Review and promotion gates

Raw ingestion and canonical promotion are deliberately separated.

Rules:

- raw ERP rows may be ingested even when canonical promotion is blocked
- if product identity is ambiguous, canonical product promotion is blocked
- if product promotion is blocked, related `prices` and `inventory` promotion is also blocked for that SKU
- review resolution is the prerequisite for downstream canonical writes

This prevents ambiguous identifiers from contaminating canonical product, pricing, or inventory state.

## Validation strategy

Validation uses both live proof and reproducible replay.

### Live validation

- use the real Oracle/Sankhya source
- begin with a bounded sample of SKUs
- validate products, prices, and inventory mappings with real data
- after confidence is established, run a full snapshot validation

### Reproducible validation

- capture fixtures from proven source queries
- build connector and promotion tests against those fixtures
- do not invent synthetic shapes that differ from verified source data

## Schema discipline

Canonical DB shape must be checked before promotion logic is written.

Rules:

- inspect existing `catalog`, `pricing`, and `inventory` schema and write paths first
- if schema changes are needed, explain why and get explicit user approval before implementing them
- prefer preserving existing canonical boundaries rather than leaking ERP-specific semantics into current modules

## Success criteria

ERP 0.1 is considered closed only when all of the following are true:

- live Oracle connectivity is proven
- source truth for `products`, `prices`, and `inventory` is verified directly from Sankhya tables
- `TGFTAB` and `TGFNTA` price-history/effective-date behavior is understood well enough to make a defensible history decision
- governed normalization is defined for price contexts and stock locations
- canonical promotion works for the validated slice
- duplicate `EAN` and duplicate manufacturer `REFERENCE` are review-gated correctly
- blocked products also block downstream price and inventory promotion
- fixture-based tests replay the validated slice reproducibly

## Risks and mitigations

### Risk: wrong source semantics

Mitigation:

- discovery before implementation
- local-only read-only workbook
- bounded live sample before full snapshot

### Risk: silent data loss from premature incremental sync

Mitigation:

- snapshot-first ingestion
- defer incremental optimization until reliable source change markers are proven

### Risk: overfitting canonical models to Sankhya naming

Mitigation:

- governed mapping for price contexts and stock locations
- preserve lineage, but normalize into MetalShopping language

### Risk: taxonomy pollution from weakly understood ERP classifications

Mitigation:

- map only the main commercial hierarchy into taxonomy in this tranche
- keep auxiliary flags raw-only until their semantics are clear
